# ADR: Smart Handler Registration for DSoftStudio.Mediator

**Status:** Rejected — Not applicable to compile-time architecture  
**Date:** 2025-07-09  
**Supersedes:** Original proposal (draft)

---

## Context

The original proposal assumed that developers would need to manually register
`INotificationHandler<T>` implementations with custom lifetimes (Singleton, Scoped,
Transient), and that calling `AddMediator()` / `RegisterMediatorHandlers()` could
cause duplicate registrations or override manual registrations.

This problem is well-documented in **MediatR** and similar mediators that use
**runtime assembly scanning** (`AddMediatR(cfg => cfg.RegisterServicesFromAssembly(...))`),
where manual registrations conflict with the scanning result.

---

## Decision

**Rejected.** The problem described does not exist in DSoftStudio.Mediator's
compile-time source-generated architecture. No implementation is needed.

---

## Rationale

### 1. Handlers Are Discovered at Compile Time — No Manual Registration Needed

The `DependencyInjectionGenerator` Roslyn source generator discovers every handler
class at build time by scanning `ClassDeclarationSyntax` nodes with `BaseList`.
The generated `MediatorServiceRegistry.Register()` emits explicit registrations for
every handler. Developers never need to call
`services.AddTransient<INotificationHandler<T>, MyHandler>()` manually.

```
Compile-time:  DependencyInjectionGenerator → scans all classes → emits MediatorServiceRegistry
Startup:       services.RegisterMediatorHandlers() → calls MediatorServiceRegistry.Register()
```

### 2. Notification Dispatch Resolves by Concrete Type, Not Interface

The high-performance default dispatch path **never calls `GetServices<INotificationHandler<T>>`**.
Instead, the generated `NotificationRegistry` populates `NotificationDispatch<T>.Handlers`
with **factory delegates that resolve by concrete type**:

```csharp
// Generated by NotificationGenerator (compile-time)
NotificationDispatch<MyNotification>.TryInitialize(
    new Func<IServiceProvider, INotificationHandler<MyNotification>>[]
    {
        static sp => sp.GetRequiredService<HandlerA>(sp),   // ← concrete type
        static sp => sp.GetRequiredService<HandlerB>(sp),   // ← concrete type
    });
```

At runtime, `NotificationCachedDispatcher.DispatchSequential()` uses these factory
delegates directly — it reads `NotificationDispatch<T>.Handlers` (the pre-built
factory array) and resolves each handler via `GetRequiredService<ConcreteType>`.

This means:
- **Duplicate interface registrations have zero effect** on the default dispatch path
- The number and order of handlers is determined at compile time, not by DI enumeration
- Even if someone adds `services.AddTransient<INotificationHandler<T>, ExtraHandler>()`,
  the dispatch table ignores it

### 3. Concrete Type Registration Already Uses `TryAdd*`

The generator emits `TryAddSingleton(typeof(ConcreteHandler))` /
`TryAddTransient(typeof(ConcreteHandler))` for concrete types:

```csharp
// Generated by DependencyInjectionGenerator
ServiceCollectionDescriptorExtensions.TryAddTransient(services, typeof(MyHandler));
//                                    ^^^^^^ — won't duplicate if already registered
```

If a developer manually registers `MyHandler` with a specific lifetime before calling
`RegisterMediatorHandlers()`, the `TryAdd*` call is a no-op — the manual registration
wins. This is already "smart" behavior without any additional code.

### 4. Handler Lifetime Is Architecturally Determined

The source generator automatically selects the optimal lifetime:

| Constructor | Lifetime | Rationale |
|-------------|----------|-----------|
| No parameters (stateless) | **Singleton** | Zero allocation per call |
| Has DI dependencies | **Transient** | Safe default, new instance per resolution |

This is the correct semantic for a mediator. Allowing arbitrary lifetime overrides
introduces risk:
- **Scoped notification handlers** + `ThreadStatic` cache = stale scope reference
  (`NotificationHandlerCache<T>` caches by `IServiceProvider` reference equality)
- **Singleton handlers with Scoped dependencies** = captive dependency bug
- **Manual Transient over auto-detected Singleton** = unnecessary allocation

### 5. `GetServices<INotificationHandler<T>>` Is Only Used in the Custom Publisher Path

The interface-based resolution only occurs in two opt-in scenarios:

| Path | Resolution | When |
|------|-----------|------|
| Default (high-performance) | `GetRequiredService<ConcreteHandler>` via factory delegates | Always |
| Custom `INotificationPublisher` | `GetServices<INotificationHandler<T>>` via DI | Only when `INotificationPublisher` is registered |
| `Publish(object)` + custom publisher | `GetServices<INotificationHandler<T>>` via DI | Only when `INotificationPublisher` is registered |

The custom publisher path is explicitly opt-in. Users who register a custom
`INotificationPublisher` are taking responsibility for their DI configuration.
Adding "smart" detection logic for this edge case adds complexity without meaningful
benefit.

### 6. The Proposed Implementation Would Add Unnecessary Startup Cost

The original plan required:
- Scanning `IServiceCollection` at registration time — **O(n)** per handler
- Building a composite key dictionary — **allocation at startup**
- Configuration option (`AllowExplicitHandlerRegistrations`) — **API surface expansion**
- Runtime detection logic — **code in the hot startup path**

For a problem that doesn't exist in the default architecture, this is pure overhead.

---

## Comparison with MediatR (Where This Problem Exists)

| Aspect | MediatR | DSoftStudio.Mediator |
|--------|---------|---------------------|
| Handler discovery | Runtime assembly scanning | **Compile-time source generation** |
| Registration method | `AddMediatR(cfg => ...)` scans + registers | `RegisterMediatorHandlers()` calls generated code |
| Notification dispatch | `GetServices<INotificationHandler<T>>` | **Factory delegates → `GetRequiredService<Concrete>`** |
| Duplicate risk | High (scan + manual = 2x) | **None** (concrete uses `TryAdd*`, dispatch ignores interface) |
| Lifetime control | Manual override needed | **Auto-detected (stateless=Singleton, stateful=Transient)** |

---

## Consequences

- **No code changes required.** The current registration architecture is already
  resilient to duplicate registrations by design.
- **Concrete types use `TryAdd*`** — manual registrations are respected if added first.
- **Dispatch tables are compile-time deterministic** — not affected by DI container state.
- **Handler lifetime is auto-detected** — no need for manual override mechanism.
- **API surface remains minimal** — no new configuration options, no new extension methods.
- **Documentation note:** README should clarify that manual handler registration is
  unnecessary and that the source generator handles everything automatically, including
  optimal lifetime selection.
